4. Contract Origination — Unhappy Path (Quarantine)

SDG Validate-Only Boundary: SmartLedger does NOT approve, override, or correct data. The originating system must fix and resubmit. The quarantine is a read-only audit trail.
sequenceDiagram participant OL as Oracle LOS (sim) participant Bus as Redis Streams participant Agent as AI Agent participant VAL as Validation Engine participant PG as PostgreSQL participant UI as Dashboard UI OL->>Bus: publish contract.originated (bad data) Bus->>Agent: deliver event Note over Agent: Acquire per-contract lock Agent->>VAL: validate_event(event + context) VAL->>VAL: INVALID_INTEREST_RATE detected VAL->>PG: INSERT validation.quarantine (status=pending) VAL-->>Agent: ValidationResult(valid=false, failures=[...]) Note over Agent: Checkpoint: QUARANTINED Note over Agent: Release lock + ACK event Note over Agent: Nothing written to ledger UI->>PG: GET /api/quarantine (read-only audit trail) Note over UI: Shows rejection reasons — no approve/override buttons Note over OL: Originating system corrects and resubmits OL->>Bus: publish contract.originated (corrected, new event_id) Note over Agent: Full validation flow runs again from scratch

4b. Customer Profile Update Flow (Integration Layer)

CRM, Portal, Mobile, and LOS push customer data changes via the Integration System. SmartLedger intercepts at this boundary to validate and audit every change before LLAS is updated.
sequenceDiagram participant SRC as Source System (CRM/Portal/Mobile/LOS) participant INT as Integration System :8022 participant Bus as Redis Streams participant Agent as AI Agent participant LLAS as LLAS Sim participant VAL as Validation Engine participant LED as Ledger MCP SRC->>INT: submit_contact_update / submit_payment_update INT->>Bus: publish integration.contact_update_requested Bus->>Agent: deliver event Note over Agent: Acquire per-contract lock Agent->>LLAS: get_customer_profile(contract_id) LLAS-->>Agent: current profile Agent->>VAL: validate_event(integration event + llas profile) VAL->>VAL: Conflict check — same field from different source? VAL->>VAL: Contract state eligibility check VAL->>VAL: Business rule + field format checks alt No conflict — valid update VAL-->>Agent: ValidationResult(valid=true, proof_token=JWT) Agent->>LED: write_record(customer_update_record, proof_token) Agent->>LLAS: update_customer_profile(contract_id, changes) Note over Agent: Checkpoint: COMPLETED else Conflict detected VAL->>VAL: Quarantine BOTH events (status=conflict, conflict_pair_id=uuid) VAL-->>Agent: ValidationResult(valid=false, code=CONFLICT_PENDING) Note over Agent: Neither update proceeds to LLAS else Invalid — rule failure VAL-->>Agent: ValidationResult(valid=false, failures=[...]) Note over Agent: Update blocked — source must fix and resubmit end Note over Agent: Release lock + ACK event

4c. Conflict Resolution Flow (LLAS Admin)

When two sources submit competing updates to the same field, both are blocked. The LLAS Admin selects the authoritative value. SmartLedger still validates before writing.
sequenceDiagram participant UI as Dashboard UI participant DA as Dashboard API participant VAL as Validation Engine participant Bus as Redis Streams participant Agent as AI Agent participant LED as Ledger MCP participant LLAS as LLAS Sim Note over UI: LLAS Admin sees Conflicts view UI->>DA: GET /api/conflicts DA-->>UI: source A value vs source B value + current LLAS profile Note over UI: Admin selects winning value + reason UI->>DA: POST /api/conflicts/{conflict_pair_id}/resolve DA->>VAL: resolve_conflict(conflict_pair_id, winning_event_id, admin_id) VAL->>VAL: Validate winning value (business rules still apply) VAL->>VAL: Issue proof token for winning event VAL->>VAL: Mark winning: status=resolved VAL->>VAL: Mark losing: status=resolved, code=CONFLICT_RESOLVED_BY_ADMIN VAL->>Bus: publish integration.conflict_resolved Bus->>Agent: deliver integration.conflict_resolved Agent->>LED: write_record(customer_update_record, proof_token) Agent->>LLAS: update_customer_profile(contract_id, winning_changes) Note over Agent: Checkpoint: COMPLETED

9. Status & State Reference

Authoritative enum values used across database, APIs, and the Governance Dashboard.

Quarantine Status

ValueSet byMeaning
pendingValidation EngineFailed validation — originating system must correct and resubmit.
conflictValidation EngineTwo sources submitted competing updates. Both quarantined as a matched pair. Awaiting LLAS Admin.
resolvedValidation EngineConflict adjudicated. Winning side applied; losing side records CONFLICT_RESOLVED_BY_ADMIN.

No approved or override status exists. SmartLedger never grants exceptions.

Contract State

ValueMeaning
originatedReceived, validated, ledger written. Pre-activation.
activeIn good standing.
delinquentPast due.
charged_offWritten off. No further updates permitted.
paid_offFully paid. Awaiting title release.
title_releasedTitle transferred. Lifecycle complete.

Integration Submission Status

ValueMeaning
pendingSubmitted to Integration System. Event published to Redis.
validatedValidated. Ledger written. LLAS updated.
quarantinedRejected. Originating system must correct and resubmit.
conflictCompeting update detected. Both quarantined pending LLAS Admin resolution.
