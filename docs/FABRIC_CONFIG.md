# SmartLedger — Hyperledger Fabric Configuration Reference

> Snapshot of the *runtime* Fabric configuration for this POC.  Maintained
> separately from the locked `REQUIREMENTS.md` (design spec) and the
> `ARCHITECTURE.md` (flows and patterns), because Fabric values are
> infrastructure-level and change independently.

**Last verified:** 2026-05-01 (Phase I complete) · **Fabric version:** `2.5.9`

## 4. Crypto Material Structure

Generated by `cryptogen` (per `crypto-config.yaml`) and persisted under
`infra/fabric/crypto-material/` — **gitignored**, regenerated by
`setup-fabric.sh` on a fresh clone.

```
crypto-material/
├── ordererOrganizations/
│   └── orderer.smartledger.local/
│       ├── msp/
│       ├── orderers/
│       │   └── orderer.orderer.smartledger.local/   ← node FQDN = host.org
│       │       ├── msp/    (orderer identity)
│       │       └── tls/    (server.crt, server.key, ca.crt)
│       └── users/Admin@orderer.smartledger.local/msp/
│
└── peerOrganizations/
    └── org1.smartledger.local/
        ├── msp/
        ├── peers/peer0.org1.smartledger.local/
        │   ├── msp/
        │   └── tls/
        └── users/Admin@org1.smartledger.local/msp/
            ├── keystore/priv_sk      ← admin private key (file literally named priv_sk)
            └── signcerts/Admin@org1.smartledger.local-cert.pem
```

**Naming quirk:** The orderer's full FQDN is `orderer.orderer.smartledger.local`
because the format is `<host>.<org>` and both happen to start with "orderer"
(`org=orderer.smartledger.local`, `host=orderer`).  The TLS SAN matches.

## 5. Capabilities Matrix

| Group | Level | Why |
|---|---|---|
| Channel | `V2_0` | Compatible with all 2.x peers/orderers |
| Orderer | `V2_0` | Lowest level supporting Raft + Channel Participation API |
| Application | `V2_5` | Required for chaincode lifecycle 2.x semantics |

If you upgrade to a higher capability level, re-running channel/config
updates is required.  Don't bump these without a plan.

## 6. Verifying Against Reality

Quick checks — run from the host (containers must be up):

| Check | Command | Expected |
|---|---|---|
| Containers running | `docker ps --format '{{.Names}}' \| grep -E 'fabric-(orderer\|peer0\|ca\|couchdb)'` | 4 lines |
| Channel exists on peer | `docker exec fabric-peer0.org1.smartledger.local-1 peer channel list` | `smartledger-channel` |
| Channel exists on orderer | `docker exec fabric-orderer.orderer.smartledger.local-1 osnadmin channel list --orderer-address orderer:7053 --ca-file /var/hyperledger/orderer/tls/ca.crt --client-cert /var/hyperledger/orderer/tls/server.crt --client-key /var/hyperledger/orderer/tls/server.key` | `smartledger-channel` |
| Chaincode committed | `docker exec fabric-peer0.org1.smartledger.local-1 peer lifecycle chaincode querycommitted -C smartledger-channel` | `smartledger-cc, Version: 1.0, Sequence: 1` |
| Block height | Look in Explorer (`http://localhost:8090`) → Network → ledger height | matches `peer channel getinfo` |
| State DB | `docker logs fabric-peer0.org1.smartledger.local-1 \| grep -i couchdb \| head -3` | references `couchdb:5984` |
| Read a record | `docker exec fabric-peer0.org1.smartledger.local-1 peer chaincode query -C smartledger-channel -n smartledger-cc -c '{"function":"QueryRecord","Args":["RECORD:<contract_id>"]}'` | JSON record |

## 7. Change Procedures

What to do when you change…

| You change | You must |
|---|---|
| **Chaincode source** (no signature change) | Bump `CC_VERSION` and `CC_SEQUENCE`, re-run `setup-fabric.sh` (which packages → installs → approves → commits). The new version coexists with the old until committed. |
| **Channel config** (e.g. capabilities, batch params) | Generate a config update transaction, sign by majority, submit via `peer channel update` |
| **Add a new org** | Update `configtx.yaml`, regenerate `crypto-material/` for the new org, perform a channel config update, install chaincode on new org's peers |
| **TLS certs / MSP** | Wipe `crypto-material/`, `orderer_data` and `peer0_data` volumes, re-run `setup-fabric.sh`. **All chain history is lost — POC only.** |
| **Fabric version** | Update `FABRIC_VERSION` in `setup-fabric.sh` and image tags in `docker-compose.fabric.yml`.  Verify capability compatibility before upgrading peers. |
| **Switch state DB to LevelDB** | Remove CouchDB env vars from peer service; remove `couchdb` service.  **Wipes world state** — must rebuild from genesis. |

**Volume wipe** (full clean restart, loses all chain data):

```bash
cd infra/fabric
docker compose -f docker-compose.fabric.yml down
docker volume rm fabric_orderer_data fabric_peer0_data
./scripts/setup-fabric.sh   # regenerates everything
```

## 8. Production-Readiness Gaps

| Gap | Current | Production target | Priority |
|---|---|---|---|
| **Orderer fault tolerance** | 1 Raft consenter | ≥3 (so 1 can fail) — odd count for quorum | High |
| **Multi-org consensus** | 1 peer org | ≥2 orgs, each running their own peer (e.g., SmartLedger, Capital Finance Corp) | High — required for SDG Path B |
| **Endorsement policy** | Implicit majority (trivial w/ 1 org) | Custom policy: e.g. `AND('SmartLedgerOrg.member','LenderOrg.member')` | High |
| **CouchDB credentials** | `admin / adminpw` | Rotated, secret-managed | High |
| **Fabric CA admin password** | `admin / adminpw` | Rotated, secret-managed; consider HSM-backed root CA | High |
| **TLS PKI** | Self-signed via `cryptogen` | Real CA with hierarchy + revocation (CRL/OCSP) | Medium |
| **Backups** | Volume-only, manual | Scheduled snapshots of orderer + peer ledger volumes; tested restore drill | Medium |
| **Monitoring** | Operations endpoints exist (`:9443`, `:9444`) but unscraped | Wire to Prometheus + Grafana; alert on fork detection, consenter unavailability | Medium |
| **Block batching tuning** | Defaults (2s, 10 msgs, 512KB) | Tune to actual throughput target | Low — defaults are fine for POC volumes |
| **Private Data Collections** | None | Per-contract collections so only the parties to a contract can read its data | Medium — alternate to multi-org for confidentiality |

## 9. File Reference

| File | What it controls |
|---|---|
| `infra/fabric/docker-compose.fabric.yml` | Service definitions (orderer, peer, CA, CouchDB, CLI) and env-driven runtime config |
| `infra/fabric/configtx.yaml` | Channel + orderer profiles, organizations, capabilities, batch params, consenters |
| `infra/fabric/crypto-config.yaml` | Input to `cryptogen` — defines orgs, peer counts, user counts |
| `infra/fabric/scripts/setup-fabric.sh` | Bootstrap: cryptogen → genesis → osnadmin join → peer join → install/approve/commit chaincode |
| `infra/fabric/scripts/start-explorer.sh` | Brings up Hyperledger Explorer against the running network |
| `infra/fabric/explorer/connection-profile/smartledger-network.json` | Explorer's view of the network (MSP, peer endpoint, cert paths) |
| `infra/chaincode/smartledger-cc/lib/smartledger-contract.js` | The chaincode — `WriteRecord`, `QueryRecord`, `ExecuteStateTransition`, `CalculateLateFee` |
| `src/mcp_servers/ledger/fabric_client.py` | Python client invoking chaincode via `peer chaincode invoke/query` subprocess |

> **Note:** `infra/fabric/config/` contains Fabric's **default sample** YAML
> files (`core.yaml`, `orderer.yaml`, `configtx.yaml`).  These are *not used*
> by the running network — they ship with the binaries for reference only.
> The active `configtx.yaml` lives at `infra/fabric/configtx.yaml`.
